Spark Application Manager
Overview

The Spark Application Manager provides a detailed view into the behavior of Spark applications. You can use it to:

  • Resolve inefficiencies, bottlenecks, and reasons for failure within applications,

  • Optimize resource allocation for Spark executors,

  • Detect and fix poor partitioning,

  • Detect and fix inefficient and failed Spark apps, and

  • Tune JVM settings for driver and executors.

There are multiple Spark applications types, for instance

  • SQL,

  • Streaming,

  • PySpark,

  • Shell, and

  • Notebook.

Currently Unravel's UI Spark Application Manager distinguishes between SQL, Streaming, and the remaining (PySpark, Shell and Notebook). The information contained with the Spark Application Manager can vary by the type of job. But, as with all Applications Managers, no matter what the job, the layout is similar and there are common tabs/information across all types.

Spark Application Managers Basic Layout

  • A black title bar notes the type of tile (Spark, Job, Stage, etc). The right side of the title bar are glyphs for adding a comment, and to minimize or close the tile if possible. If it has a link to the parent, there will be an up arrow () on the right hand side.

  • Unravel's Intelligence Engine provides insights into an application and may provide recommendations, suggestions or insights into how to improve the application's run. When there are insights a bar appears immediately below the title bar. If Unravel has recommendations the insight bar is orange, otherwise it's blue. For more information about events, see the Event Panel Examples in the Applications Tab.

  • The next section contains the Key Performance Indicators (KPIs) and general job information.

    • Event icon: notes the number of events the job had. If there were no events No Events is noted instead of the box. This job has 4 events, clicking on the icon brings up a panel which contains one or more sub-tabs, as relevant, i.e., Recommendations, Efficiencies, Application Failure. See the Event Panel Examples for further information.

    • Job icon: notes the job's status and the window type (S-Spark, SJ-Spark Job and so on.). The box is colored code the same as the application status.

    • Job information: type of Spark job, user, queue, start and stop time, etc.

    • Key Performance Indicators (KPIs): vary by job type. There are none listed for a streaming job.

  • The last section, typically divided into two, has specific information related to the application/job. The sections for a specific Spark Application (e.g Streaming) go into more detail. If the job is composed of tasks/jobs/stages they appear on the the left side under Navigation. Clicking a row brings up detailed information about it in a separate tile/panel below. If there is an Auto Action column () the number of events associated with the stage is listed.

Spark Application Managers

Regardless of the type of Spark job, every Application Manager view is split into two sides.

Every application type has the following three tabs on the left:

  • Errors: Lists all errors associated with the job. The errors are color coded and the number for each type (fatal, errors, warnings) noted. The total number of errors is listed at the top. The errors are grouped by executors with the number and types of errors noted. Errors are categorized by severity, and include keywords and details associated with each. Keywords help to extract important details from the error messages/log data to help quickly root out the cause of a problem. If there are no errors "No errors found" is listed.

  • Logs: List the available logs. It is noted if no logs are available. Click on the log name to view its information.

    Below is a excerpt of the executor-20 log.

  • Conf: Lists the configuration parameters and their values. The parameters vary according to app/task/job.

    When choosing a configuration item to display, it is possible that some configurations parameters apply to other configurations. If that is the case, the applicable configurations are highlighted. In the example below, memory was selected. The memory's configuration also contains parameters relevant to: driver (spark.driver.memory), executor (spark.executor.memory) and resources (yarn.nodemanager.resource.memory-mb).

Every job has the following four tabs on the right:

  • Program: The information in this tab varies by job. The sections on specific Spark types go into further detail.

  • Task Attempts: Graphically and textually notes the number of tasks and their status.

    The donut graphs shows the percentage of successful (green) and of failed (orange) tasks. The legend on the right lists the number of successes or failure. The graph on the left shows a job in which all tasks succeeded, while the graph on the right has all failed tasks. Frequently, the result is a combination of failed and successful jobs. Hovering over the chart tells you the percentage of each.

  • Graphs: There are 4 types of graphs. Graph metrics are graphed over "wall clock time" as opposed to computer usage time. You can chose how to display the graph, the default is a line graph.

    • Attempt: The slot usage.

    • Containers: The number of running containers.

    • Vcores: Allocated vcores.

      Memory: Allocated memory.

    • Resource: Graphs the resources the application/job/stage consumed.

      By default the Resource tab graphs using the metric systemCpuLoad for the the available series/executors up to ten. You chose one or more series to display in the Select box (1). You can select the number of series to display from one to total number of series (2). Clicking on a series name toggles the view (adds or removes it from the graph); double clicking the name graphs that series alone. You chose the Metric to graph using the pull down menu. The metrics fall into one of the following categories: CPU, OS Memory, Java Memory, Java GC, and Java GC (optional). Click Get Data to retrieve the metric's data which can be viewed in its Raw form, JSON, or as headers.

Spark Streaming
Enabling Spark Streaming

The Spark Streaming probe is disabled by default and you must enable it manually by editing spark-defaults.conf. Please note that currently we support the spark streaming feature for Spark 1.6.x.

After the Unravel sensor has been deployed and installed on the cluster open spark-defaults.conf and search for spark.driver.extraJavaOptions. Set the property to -javaagent:${UNRAVEL_SENSOR_PATH}/btrace-agent.jar=script=DriverProbe.class:SQLProbe.class:StreamingProbe.class,libs=spark-1.6.

Key Performance Indicators

Unlike other Spark applications the Spark-Streaming APM lists no KPI's other than:

  • Events: is the number, if any, of Unravel insights for the query. See the Event Panel Examples for more information.

The Application Manager has a Stream tab and opens using that view on the left and Graph: Attempts on the right.

The left tabs are:Stream Tab

This tab displays the core of an Streaming Application and from here you drill down into the batches, the main processing unit for Spark streaming.

The graph displays the number of events/second (the bars) with a superimposed line graph of the chosen metric. By default Total Delay is used. Use the Metric pulldown menu to change the metric to Processing Time or Scheduling Delay. In the example above, the graph is displaying Processing Time.

This graph is composed of two sections. By default, the upper and lower section (1) display the entire run. The table lists the stream batches relevant to the time period selected in the lower section of the graph as shown in the screenshot below. The table lists only the first three batches, but you can page through the table (2). By default the streams are sorted on start time in ascending order. The batch table is initially sorted by time in ascending order and lists KPIs for the batches, e.g., Input Records, Scheduling/Processing/Total Delaty and the number of Stream operations.

You can chose the time window to display by clicking and dragging over a region of the lower section of the graph. You can expand/contract the section to display by pulling/pushing the edge () or move the box by clicking on it and sliding it to the new location ().

The Stream Batch table is updated to reflect the time period selected. In this example the right three batch groups were selected from the graph above. In the Spark APM window above there is 43 pages of stream batches; with the narrowed time selection there are now only 10 pages of streams.

In the example below, the right most 3 batches were selected, note the batches have decreased from 43 to 13.

Clicking on a batch brings up the Spark Stream Batch. The batch window lists all the jobs associated with the batch and the batch's metadata.

The title bar notes it's a Spark Stream Batch view and that it's part of a Spark Streaming application. It has two tabs, Output Operations and Metadata.The KPI's, duration, processing delay, scheduling delay, and total delay are listed.The default view is Output Operations which contains a list of outputs descriptions, the total duration, the Job ID of all jobs of the batch with the duration of each job and its status. In the example below, the first output has three jobs associated with it (the bracketed jobs), the second output only one.

The Metadata tabs lists metadata metrics and their values.

Clicking on a job id brings up the Spark Job window in the same block thereby replacing the Spark Stream Batch information.

The title bar notes it is a Spark Job and part of a Spark Streaming application. In the title bar there is an up arrow glyph (), click on it to go to the job's parent. In this case the parent is the Spark Stream Batch (shown above). The KPIs listed are Duration and # of Stages. The two tabs are Stages (the default view, and Metadata. The table lists the stages along with their KPIs. The Tasks column notes the "number of tasks completed" over the "total number of tasks".

The metadata and its associated values are shown. Click on the Batch ID value to return to the Spark Stream Batch block.

Click on the Stage to bring up the Spark Stage panel. The Spark Stage is displayed in the same block thereby replacing the Spark Job information. The Spark Stage view is not specific to the job type. See Stage Details for a comprehensive look at the stage information.

Spark Stage Details

The Spark Stage block displays two KPIs, duration and Data IO; it has two left tabs (Graphs and Timeline) and one right tab (Attempts). The Graph tab in turn has two sub tabs, Task Attempts and Program. If the Stage has a "parent" application an arrow () will appear on the right side of the title bar. Clicking on it will bring you back to the stage's application.

The default view is Graph Task Attempts and Attempts. Task Attempts has a donut chart which graphically shows the percentage of successful and failed tasks. Attempts displays the Slot usage over "wall clock" time. Above the graph the computer time is noted. In the example below "wall clock" time equals computer time.

Program Details

Clicking on the Source File Name or Call Site Line Number brings up the Spark APM. There is a description of the program details, for instance org.apache.spark.streaming.dstream.DStream.print(DStream.scala:757).

Timeline Block

The Timeline tab has two sections,

  • Distribution charts, and

  • Timeline, Timeline Breakdown and Selected Tasks.

Distribution Charts

The Distribution Charts graph Map Tasks over ShuffleMap statistics (Seconds, Input KB, and Output KB) graph. Click the radio button to switch the Distribution Chart to map over byte information (Disk Bytes Spilled, Memory Bytes Spilled, and Records Read). The Timeline graph reflects the series relevant to the time period selected in the Distribution Chart. By default the entire timeline is selected and all executors are displayed.

You can chose the time period to display by clicking and dragging to select a time period. You can expand/contract the section by pulling/pushing the edge, or move the box by clicking on it and sliding it to the new location. Only the series relevant to that time period are displayed. In this example, the time period from 24-48 seconds has been selected, during which no series was running, so the Timeline is empty.

Timeline

The Timeline Tab has three sub-tabs: Timeline, Timeline Breakdown and Selected Tasks.

  • Timeline

    The timeline is a Gantt chart of the series which makes up the stage. See Distribution Charts above for an example of the graph. Only the series relevant to the time selected are displayed. You can filter the display by Tasks, Killed/Failed, or Show All. Hovering over the timeline brings up the the KPI's for the series listed.

    Example Text Box

  • Timeline Breakdown

    This is useful to identify bottlenecks. For each executor used in the current stage multiple metrics are graphed: Scheduler Delay, Executor Deserialization Time, Fetch Wait Time, Executor Computing Time, JVM GC time, Result Serialization Time and Getting Result Time. Using the ratio of Executor Computing Time to the total execution time you can determine where the executor spent most of it time:

    performing actual work

    thrashing, or

    waiting for scheduling.

    In this example, there is only one executor.

  • Selected Tasks

A list of tasks, if any, for the stage.
